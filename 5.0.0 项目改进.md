# 多门店 ERP 系统 v5.0.0 项目改进文档

## 1. 需求概述

本次改进主要涉及以下4个方面：

1. **供货商管理**：记录供货商的基本信息、对公信息以及联系人信息
2. **商品管理**：去掉库存数量与店铺字段
3. **库存管理**：新增入库操作流程，支持选择供应商、品牌、型号搜索商品，支持串码自动生成或手动录入，待入库列表核对确认后入库
4. **串号全生命周期管理**：确保串号全系统唯一，记录所有串号操作日志，支持溯源追踪

## 2. 数据模型变更

### 2.1 新增供货商模型 (Supplier)

```prisma
model Supplier {
  id          String   @id @default(cuid())
  name        String   // 供货商名称
  code        String   @unique // 供货商编码
  address     String?  // 地址
  phone       String?  // 联系电话
  email       String?  // 邮箱
  // 对公信息
  companyName String?  // 公司名称
  taxNumber   String?  // 税号
  bankName    String?  // 开户银行
  bankAccount String?  // 银行账号
  // 联系人信息
  contactName    String?  // 联系人姓名
  contactPhone   String?  // 联系人电话
  contactEmail   String?  // 联系人邮箱
  contactPosition String? // 联系人职位
  // 状态
  status       String   @default("active") // active, inactive
  notes        String?  // 备注
  shopId       String
  shop         Shop     @relation(fields: [shopId], references: [id], onDelete: Cascade)
  purchaseOrders PurchaseOrder[]
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  @@index([shopId, status])
  @@index([name])
  @@index([code])
}
```

model PurchaseOrder {

### 2.3 修改商品模型 (Product)

```prisma
model Product {
  // ... 现有字段

  // 移除 shopId 字段（商品将跨店铺共享）

  // ... 其他字段保持不变

```

### 2.4 新增入库单模型 (StockInOrder)

```prisma
model StockInOrder {
  id          String          @id @default(cuid())
  shopId      String
  shop        Shop            @relation(fields: [shopId], references: [id], onDelete: Cascade)
  supplierId  String?
  supplier    Supplier?       @relation(fields: [supplierId], references: [id], onDelete: SetNull)
  warehouseId String?
  warehouse   Warehouse?      @relation(fields: [warehouseId], references: [id], onDelete: SetNull)
  operatorId  String
  operator    User            @relation(fields: [operatorId], references: [id], onDelete: Cascade)
  status      String          @default("pending") // pending, confirmed, completed, cancelled
  totalItems  Int             @default(0)
  notes       String?
  createdAt   DateTime        @default(now())
  updatedAt   DateTime        @updatedAt
  items       StockInOrderItem[]
  logs        StockInOrderLog[]
}

model StockInOrderItem {
  id               String         @id @default(cuid())
  orderId          String
  order            StockInOrder   @relation(fields: [orderId], references: [id], onDelete: Cascade)
  productId        String
  product          Product        @relation(fields: [productId], references: [id], onDelete: Cascade)
  quantity         Int
  price            Decimal
  serialNumberType String         // manual, auto // 串号生成方式
  serialNumbers    SerialNumber[]
  createdAt        DateTime       @default(now())
}

model StockInOrderLog {
  id          String       @id @default(cuid())
  orderId     String
  order       StockInOrder @relation(fields: [orderId], references: [id], onDelete: Cascade)
  oldStatus   String
  newStatus   String
  reason      String
  operatorId  String
  operator    User         @relation(fields: [operatorId], references: [id], onDelete: Cascade)
  createdAt   DateTime     @default(now())
}
```

### 2.5 增强串号模型 (SerialNumber)

model SerialNumber {

## 3. 对系统现有功能的影响

### 3.1 供货商管理

**影响范围：**
- 新增供货商管理页面
- 新增供货商API路由
- 采购订单需要关联供货商
- 菜单配置需要添加供货商管理入口

**影响评估：**
- ✅ 低风险：新增功能，不影响现有功能
- 🟡 中风险：采购订单需要关联供货商，需要修改现有采购订单流程

### 3.2 商品管理

**影响范围：**
- 商品列表页需要隐藏库存数量和店铺列
- 商品表单需要移除库存数量和店铺字段
- 商品模型需要调整字段（可选，保留用于兼容）
- 商品API需要相应调整

**影响评估：**
- 🔴 高风险：涉及现有商品数据的展示和管理
- 🟡 中风险：库存逻辑需要从商品转移到串号管理
- ✅ 建议：采用渐进式迁移，先隐藏字段，保留数据兼容

### 3.3 库存管理

**影响范围：**
- 新增入库单管理页面
- 新增入库单API路由
- 库存列表需要添加入库操作入口
- 需要实现商品搜索功能（按供应商、品牌、型号）
- 需要实现串号生成逻辑（手动/自动）
- 需要实现待入库列表核对功能

**影响评估：**
- 🔴 高风险：核心业务流程变更
- 🟡 中风险：涉及多个新页面和复杂交互
- ✅ 建议：分阶段实现，先实现基础入库流程

### 3.4 串号管理

**影响范围：**
- 确保串号唯一性约束
- 需要在所有串号操作处记录日志
- 新增串号溯源页面
- 需要完善串号状态流转逻辑

**影响评估：**
- 🔴 高风险：涉及现有串号数据的完整性
- 🟡 中风险：需要在多处添加日志记录
- ✅ 建议：先实现唯一性约束，再完善日志记录

## 5. 建议与不足分析

### 5.1 建议

#### 5.1.1 实施策略建议

**分阶段实施：**
1. **第一阶段**：实现供货商管理和商品管理调整
   - 风险较低，可以独立实施
   - 为后续功能奠定基础

2. **第二阶段**：实现入库单基础功能
   - 实现商品搜索和基本入库流程
   - 暂不实现串号管理

3. **第三阶段**：实现串号管理和溯源
   - 实现串号生成和唯一性检查
   - 实现串号日志记录和溯源

4. **第四阶段**：完善和优化
   - 集成测试
   - 性能优化
   - 用户体验优化

#### 5.1.2 技术建议

**数据迁移策略：**
- 商品模型采用渐进式迁移，先隐藏字段，保留数据兼容
- 库存数量可以继续显示，但实际库存由串号管理
- 添加数据迁移脚本，便于后续完全迁移

**串号生成策略：**
- 自动生成串号建议采用有意义的格式，如：品牌缩写+日期+序列号
- 可以配置串号生成规则
- 建议添加串号前缀和后缀配置

**日志记录策略：**
- 建议使用统一的日志记录工具类
- 日志应该包含：操作人、操作时间、操作类型、变更前后状态、备注
- 建议添加日志归档功能

#### 5.1.3 用户体验建议

**入库流程优化：**
- 支持扫码枪录入串号
- 支持Excel批量导入商品和串号
- 添加智能提示，如重复串号警告
- 支持保存草稿，下次继续编辑

**串号溯源优化：**
- 时间轴展示串号生命周期
- 支持导出串号溯源报告
- 添加串号状态流转图
- 支持按时间段筛选操作日志

### 5.2 不足分析

#### 5.2.1 当前系统不足

1. **缺少供货商管理**
   - 现状：采购订单没有供货商信息
   - 影响：无法统计供货商采购数据，无法管理供货商信息
   - 改进：新增供货商管理模块

2. **商品管理包含库存和店铺**
   - 现状：商品模型包含库存数量和店铺字段
   - 影响：库存管理不够精细，商品无法跨店铺共享
   - 改进：移除这些字段，由串号管理库存

3. **入库流程不够完善**
   - 现状：只有简单的库存调整功能
   - 影响：无法管理入库流程，无法核对入库商品
   - 改进：新增入库单管理，支持待入库列表核对

4. **串号管理不够完善**
   - 现状：串号有基本状态管理，但缺少完整的日志记录
   - 影响：无法追溯串号的完整生命周期
   - 改进：完善串号日志记录，实现溯源功能

#### 5.2.2 本次改进可能存在的不足

1. **数据迁移风险**
   - 商品模型调整可能影响现有数据
   - 建议：充分测试，保留数据兼容

2. **性能影响**
   - 串号唯一性检查可能影响性能
   - 建议：添加索引，批量操作时优化

3. **复杂度增加**
   - 入库流程变得复杂，用户学习成本增加
   - 建议：提供详细的使用文档和教程

4. **测试覆盖不足**
   - 新增功能较多，测试工作量大
   - 建议：优先测试核心流程，逐步完善测试

## 6. 总结

本次v5.0.0改进是一次重要的系统升级，主要解决了供货商管理、商品管理、库存管理和串号管理四个方面的问题。虽然改进涉及多个模块，风险较高，但通过分阶段实施和充分的测试，可以有效降低风险。

**核心改进点：**
1. 完善的供货商管理，支持基本信息、对公信息和联系人信息
2. 商品管理去库存化，实现商品跨店铺共享
3. 完善的入库流程，支持商品搜索、串号管理和待入库核对
4. 完整的串号生命周期管理，支持溯源追踪

**预期收益：**
- 提升供货商管理效率
- 实现更精细的库存管理
- 提高串号管理的可追溯性
- 为未来的数据分析和报表功能奠定基础

**风险提示：**
- 数据迁移风险
- 性能影响
- 用户学习成本
- 测试工作量大

建议按照分阶段实施策略，逐步推进改进，确保系统稳定运行。

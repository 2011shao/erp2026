### 改进背景

当前系统存在以下问题：
- ❌ 后端无法自由添加角色与权限
- ❌ 无法对不同身份进行左侧菜单的控制
- ❌ 权限控制硬编码，缺乏灵活性
- ❌ 前端菜单固定显示，不基于用户权限

本方案旨在解决这些问题，实现灵活的角色权限管理和基于权限的前端菜单控制。

## 2. 核心改进内容

### 2.1 权限模型设计

#### 2.1.1 权限模型

**权限层级：**
1. **功能权限**：控制用户是否可以访问某个功能模块
2. **操作权限**：控制用户是否可以执行特定操作（如添加、编辑、删除）
3. **数据权限**：控制用户可以访问的数据范围（如只能访问本店铺数据）

**权限对象：**
- **菜单权限**：控制左侧菜单的显示
- **API权限**：控制API接口的访问
- **操作权限**：控制页面操作按钮的显示

#### 2.1.2 角色模型

**内置角色：**
- `admin`：超级管理员，拥有所有权限
- `manager`：店长，拥有本店铺的所有权限
- `staff`：员工，拥有基础操作权限

**自定义角色：**
- 支持管理员创建自定义角色
- 为角色分配具体权限

### 2.2 后端实现

#### 2.2.1 数据库模型调整

**新增模型：**
- `Role`：角色表
- `Permission`：权限表
- `RolePermission`：角色权限关联表
- `UserRole`：用户角色关联表（可选，支持用户多角色）

**数据结构：**
```prisma
// 权限表
model Permission {
  id          String   @id @default(cuid())
  code        String   @unique // 权限编码
  name        String   // 权限名称
  description String?  // 权限描述
  type        String   // 权限类型：menu, api, action
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  roles       Role[]   @relation(through: RolePermission, references: [id])
}

// 角色表
model Role {
  id          String   @id @default(cuid())
  name        String   @unique // 角色名称
  description String?  // 角色描述
  isSystem    Boolean  @default(false) // 是否系统内置角色
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  permissions Permission[] @relation(through: RolePermission, references: [id])
  users       User[]   @relation(through: UserRole, references: [id])
}

// 角色权限关联表
model RolePermission {
  roleId       String   @id
  permissionId String   @id
  role         Role     @relation(fields: [roleId], references: [id])
  permission   Permission @relation(fields: [permissionId], references: [id])
  @@id([roleId, permissionId])
}

// 用户角色关联表
model UserRole {
  userId String @id
  roleId String @id
  user   User   @relation(fields: [userId], references: [id])
  role   Role   @relation(fields: [roleId], references: [id])
  @@id([userId, roleId])
}
```

#### 2.2.2 API 实现

**新增 API 端点：**
- `GET /api/roles` - 获取角色列表
- `GET /api/roles/:id` - 获取角色详情（含权限）
- `POST /api/roles` - 创建角色
- `PUT /api/roles/:id` - 更新角色
- `DELETE /api/roles/:id` - 删除角色
- `GET /api/permissions` - 获取权限列表
- `GET /api/permissions/:id` - 获取权限详情
- `POST /api/roles/:id/permissions` - 为角色分配权限
- `DELETE /api/roles/:id/permissions/:permissionId` - 移除角色权限
- `GET /api/users/:id/roles` - 获取用户角色
- `POST /api/users/:id/roles` - 为用户分配角色
- `DELETE /api/users/:id/roles/:roleId` - 移除用户角色

**权限验证改进：**
- 实现基于权限代码的验证中间件
- 支持细粒度的权限控制

### 2.3 前端实现

#### 2.3.1 菜单权限控制

**实现方案：**
1. **菜单配置**：定义菜单与权限的对应关系
2. **权限获取**：登录时获取用户权限列表
3. **菜单过滤**：根据用户权限动态生成菜单
4. **权限缓存**：缓存用户权限信息

**菜单配置示例：**
```typescript
const menuConfig = [
  {
    key: '1',
    icon: <HomeOutlined />,
    label: '首页',
    path: '/',
    permission: 'dashboard.view'
  },
  {
    key: '2',
    icon: <ShopOutlined />,
    label: '店铺管理',
    path: '/shops',
    permission: 'shop.manage',
    children: [
      {
        key: '2-1',
        label: '店铺列表',
        path: '/shops',
        permission: 'shop.view'
      },
      {
        key: '2-2',
        label: '添加店铺',
        path: '/shops/add',
        permission: 'shop.create'
      }
    ]
  }
];
```

#### 2.3.2 操作权限控制

**实现方案：**
1. **按钮权限**：根据权限控制按钮显示
2. **路由权限**：权限不足时重定向
3. **权限检查**：提供权限检查工具函数

**权限检查示例：**
```typescript
const hasPermission = (permissionCode) => {
  const userPermissions = useAuthStore(state => state.permissions);
  return userPermissions.includes(permissionCode);
};

// 使用
{hasPermission('shop.create') && (
  <Button type="primary" icon={<PlusOutlined />}>添加店铺</Button>
)}
```

1. **权限数据迁移**：现有用户的权限分配
2. **性能影响**：权限检查对系统性能的影响
3. **兼容性**：与现有系统的兼容性
4. **复杂度**：权限管理的复杂度增加

### 7.2 应对措施

1. **权限迁移**：制定迁移方案，确保现有用户权限不受影响
2. **性能优化**：权限缓存和优化权限检查逻辑
3. **兼容性**：保持向后兼容，渐进式实现
4. **文档完善**：提供详细的权限管理文档

## 8. 总结

本改进方案通过实现灵活的角色权限管理和基于权限的前端菜单控制，解决了当前系统中权限管理僵化的问题。通过引入权限模型、角色管理和前端权限控制，系统将变得更加安全、灵活和可维护。

实施本方案后，管理员可以：
- 创建自定义角色
- 为角色分配细粒度权限
- 控制不同用户的菜单显示
- 实现基于权限的操作控制

这将大大提升系统的安全性和管理灵活性，为后续的功能扩展和系统维护奠定基础。
